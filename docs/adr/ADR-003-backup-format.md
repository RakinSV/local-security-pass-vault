# ADR-003: Формат файла бэкапа

**Статус**: Принято  
**Дата**: 2026-06  
**Контекст**: core-vault/backup

## Контекст

Нужен формат для зашифрованных бэкапов vault.db, который:
- Защищён даже если файл утечёт публично
- Не требует инфраструктуры (просто файл)
- Открыт для аудита (не самописный крипто-формат)
- Поддерживает верификацию целостности без расшифровки

## Рассматривали

| Вариант | За | Против |
|---------|-----|--------|
| GPG (PGP) | Распространён, CLI доступен | Сложный, legacy, UX плохой |
| Самописный формат | Полный контроль | Нельзя аудировать, легко ошибиться |
| age | Современный, простой, аудированный | Меньше известен |
| Encrypted ZIP | Знаком пользователям | ZipCrypto слабый, AES-ZIP сложно правильно |

## Решение

**age** (rust crate: age = "0.10") + **BLAKE3** checksum внутри payload.

Ключ получается из BIP-39 мнемоники (24 слова, 256 бит энтропии) через
Argon2id с усиленным профилем (4GB RAM, 10 итераций).
Этот профиль KDF делает офлайн-брутфорс практически невозможным даже
при компрометации файла бэкапа.

## Последствия

- Зависимость на `age`, `bip39`, `blake3` crates
- Пользователь должен хранить 24-словную мнемонику (UX задача)
- Восстановление требует мнемоники — без неё бэкап нерасшифруем
  (это намеренное свойство, не баг)
- Совместимость: файл .vbk можно расшифровать внешним инструментом
  age CLI при наличии ключа
- Размер бэкапа ≈ размер vault.db + ~1 КБ overhead

## Ревизия

Пересмотреть если: появится стандарт для зашифрованных бэкапов
менеджеров паролей (например KDBX-подобный для CLI-инструментов).
