# ADR-001: Выбор криптографической библиотеки

**Статус:** Принято  
**Дата:** 2026-06  
**Контекст:** core-vault

## Контекст

Нужна кроссплатформенная криптографическая библиотека для Rust,
покрывающая: KDF (Argon2id), AEAD шифрование (XChaCha20-Poly1305),
защищённую работу с памятью (mlock, memzero), CSPRNG.

## Решение

**libsodium через sodiumoxide/libsodium-sys** со статической линковкой.

## Альтернативы

| Критерий | libsodium | ring | RustCrypto | BouncyCastle |
|----------|-----------|------|------------|--------------|
| Независимый аудит | ✅ Многократный | ✅ Да | ⚠️ Частично | ✅ Да |
| Argon2id встроен | ✅ Да | ❌ Нет | ✅ Отдельный crate | ✅ Да |
| Secure memory API | ✅ mlock+memzero | ❌ Нет | ❌ Нет | ❌ Нет |
| XChaCha20 (192-bit nonce) | ✅ Да | ❌ Нет | ✅ Да | ⚠️ Не стандарт |
| Статическая линковка | ✅ Да | ✅ Да | ✅ Да | ❌ JVM |
| Кроссплатформенность | ✅ Win/Lin/Mac/iOS | ✅ Да | ✅ Да | ❌ Только JVM |
| Язык | C (биндинги) | Rust | Rust | Java |

## Обоснование выбора

libsodium — единственная библиотека где всё необходимое собрано в одном
аудированном пакете. Secure memory API (mlock/memzero) критичен для нас
и отсутствует в других вариантах. Написана на C, но аудит прошёл именно C код
(не биндинги) — это лучше чем молодые Rust-реализации без аудита.

Статическая линковка устраняет риск DLL hijacking.

## Последствия

- Бинарь увеличится на ~500 КБ (libsodium статически)
- Нет зависимости от системного openssl/libsodium — предсказуемое поведение
- При появлении критической CVE в libsodium — нужен новый релиз (не просто обновление пакета)
- Сложнее заменить алгоритм в будущем (чем если бы использовали модульный RustCrypto)

## Ревизия

Пересмотреть если: RustCrypto crates пройдут независимый аудит включая Argon2id + добавят secure memory API.
